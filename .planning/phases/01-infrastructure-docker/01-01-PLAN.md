<task type="auto" tdd="false">
  <name>Task 2: Write Dockerfile layering DuckDB + R packages + Python stack</name>
  <files>Dockerfile, .dockerignore</files>
  <action>
Create `Dockerfile` at project root based on `quay.io/jupyter/datascience-notebook` with a dated tag (per RESEARCH.md: use `2026-03-30` — the dated tag, NOT `:latest`, for reproducibility).

Dockerfile content:

```dockerfile
# DoML analysis environment — Python 3.13 + R 4.x + DuckDB + ML stack
# Base: Jupyter DataScience Notebook (quay.io — Docker Hub frozen since 2023-10-20)
FROM quay.io/jupyter/datascience-notebook:2026-03-30

LABEL maintainer="DoML framework"
LABEL description="Reproducible ML analysis environment with Python, R, and DuckDB"

# --- R packages (single RUN layer to minimize Docker layers) ---
# Runs as root to install into system R library, then reverts to jovyan.
USER root
RUN Rscript -e 'install.packages(c( \
      "duckdb", \
      "arrow", \
      "tidymodels", \
      "renv", \
      "prophet", \
      "forecast", \
      "umap" \
    ), repos="https://cloud.r-project.org", dependencies=c("Depends","Imports","LinkingTo"), Ncpus=parallel::detectCores())' && \
    fix-permissions "${R_LIBS_USER}" && \
    fix-permissions "/opt/conda/lib/R/library"

# --- Python packages (pinned via requirements.txt) ---
USER ${NB_UID}
COPY --chown=${NB_UID}:${NB_GID} requirements.txt /tmp/requirements.txt
RUN pip install --no-cache-dir --requirement /tmp/requirements.txt && \
    fix-permissions "${CONDA_DIR}" && \
    fix-permissions "/home/${NB_USER}"

# --- DoML working directory ---
ENV PROJECT_ROOT=/home/jovyan/work
WORKDIR ${PROJECT_ROOT}
```

Also create `.dockerignore` at project root:

```
.git
.gitignore
*.pyc
__pycache__
.ipynb_checkpoints
.pytest_cache
.venv
venv
renv/library
data/raw/*
data/processed/*
data/external/*
!data/**/.gitkeep
reports/*.html
models/*.pkl
.DS_Store
```

Rationale per RESEARCH.md:
- `USER root` for R installs → `fix-permissions` → `USER ${NB_UID}` is the documented jupyter-docker-stacks pattern
- `Ncpus=parallel::detectCores()` parallelizes R package C++ compilation (prophet, arrow benefit most)
- `dependencies=c("Depends","Imports","LinkingTo")` (not `TRUE`) keeps image leaner by skipping Suggests
- `pip install --no-cache-dir` is mandatory per the constraint
- Single R RUN layer = minimal Docker layers per constraint
- `PROJECT_ROOT` env var satisfies REPR-02 (no hardcoded absolute paths in notebooks)

<task type="auto" tdd="false">
  <name>Task 3: Write docker-compose.yml with read-only raw data mount</name>
  <files>docker-compose.yml</files>
  <action>
Create `docker-compose.yml` at project root. This is the ONLY entry point end users run — `docker compose up` and the environment is live.

Content:

```yaml
services:
  jupyter:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: ${PROJECT_NAME:-doml}-jupyter
    ports:
      - "8888:8888"
    environment:
      - JUPYTER_ENABLE_LAB=yes
      - JUPYTER_TOKEN=${JUPYTER_TOKEN:-}
      - PROJECT_ROOT=/home/jovyan/work
      - CHOWN_HOME=yes
      - CHOWN_HOME_OPTS=-R
    volumes:
      # INFR-05: raw + external data mounted read-only — immutability enforced at OS level
      - ./data/raw:/home/jovyan/work/data/raw:ro
      - ./data/external:/home/jovyan/work/data/external:ro
      # Writable mounts — analysis outputs
      - ./data/processed:/home/jovyan/work/data/processed
      - ./notebooks:/home/jovyan/work/notebooks
      - ./reports:/home/jovyan/work/reports
      - ./models:/home/jovyan/work/models
    working_dir: /home/jovyan/work
    user: root
    command: start-notebook.py --NotebookApp.token='${JUPYTER_TOKEN:-}'
    restart: unless-stopped
```

Key decisions (from RESEARCH.md):
- Port `8888:8888` satisfies INFR-02 localhost:8888 requirement
- `./data/raw:...:ro` and `./data/external:...:ro` enforce INFR-05 immutability at the Docker layer (OS-level, strongest guarantee)
- `PROJECT_ROOT=/home/jovyan/work` env var satisfies REPR-02
- `CHOWN_HOME=yes` repairs volume ownership so jovyan can write — prevents host-UID permission issues
- `user: root` is required by jupyter image's init scripts; the image's entrypoint drops to jovyan automatically after init
- Empty `JUPYTER_TOKEN` default is fine for local dev; override via env for shared use
- No `image:` key — always builds from local Dockerfile to include DoML stack
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → container | Host filesystem bind-mounted into container |
| container → host | Container processes can write to mounted volumes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | data/raw volume | mitigate | Bind mount uses `:ro` flag — OS-level read-only enforcement per INFR-05 |
| T-01-02 | Tampering | Python dependencies | mitigate | requirements.txt pins all versions with `==` — prevents supply-chain drift via unpinned deps |
| T-01-03 | Information Disclosure | Credentials in image | mitigate | No secrets in Dockerfile or docker-compose.yml; `JUPYTER_TOKEN` via env var only |
| T-01-04 | Elevation of Privilege | Container user | mitigate | Image drops to `jovyan` (NB_UID) after init; non-root by default |
| T-01-05 | Tampering | Base image integrity | accept | Dated tag `2026-03-30` on quay.io is immutable; acceptable trust in jupyter project |
</threat_model>
