## Decision 1: Notebook Template vs. Dynamic Generation

**Method:** Static nbformat v4 template file at `.claude/doml/templates/notebooks/business_understanding.ipynb`.

The template is a fully valid Jupyter notebook. It reads `.planning/config.json` and `.planning/PROJECT.md` at execution time via Python code cells using `PROJECT_ROOT` (REPR-02). All content is dynamic — no Jinja2 rendering of the template file itself.

**Why not Jinja2 template rendering?** The notebook JSON contains escaped quotes and braces that are awkward in Jinja2. Dynamic content via Python code cells is cleaner and produces a notebook that's also runnable standalone inside Docker without extra tooling.

**Why a static template file (not fully generated by Claude)?** The template is reusable across all analysis projects. It's committed to the framework repo and copied into any project that uses DoML. This ensures consistent structure and makes review/testing possible without running the full pipeline.

---

## Decision 2: Execution Mechanism — nbconvert --execute (not papermill)

**Method:** `jupyter nbconvert --execute --to notebook --inplace notebooks/01_business_understanding.ipynb`

Papermill adds value when injecting parameters that vary run-to-run (e.g., date ranges, model hyperparameters). For BU, ALL content is read from project files at runtime — no injection needed. The template reads config.json and PROJECT.md itself.

**Papermill compatibility preserved:** The template includes a `# Parameters` cell tagged with the `parameters` metadata tag so papermill CAN be used if needed, but the default execution path uses nbconvert.

**Docker execution pattern:** Run via `docker compose exec jupyter` — same pattern as existing project conventions. Claude Code runs the Bash command using `docker compose exec jupyter jupyter nbconvert --execute ...`.

---

## Decision 3: Executive Narrative Approach

**Method:** Claude generates a narrative Markdown cell and writes it into the notebook BEFORE HTML conversion.

After the notebook is executed, Claude reads PROJECT.md and the executed notebook to understand the findings, then:
1. Writes a 2–3 paragraph executive summary as plain Markdown
2. Uses `nbformat` (via a Python script) to prepend this as a Markdown cell to the executed notebook
3. Runs `jupyter nbconvert --to html --no-input` to generate HTML

This means the executive narrative appears in both the notebook and the HTML report. The narrative is written by Claude using its language model capabilities — not generated by a Python function.

**Caveats cell:** The template includes a final Markdown cell with the required caveats (OUT-03), including the explicit "Correlation is not causation" disclaimer. This is static text in the template — always present.

## Decision 5: Data Inventory DuckDB Queries

**Content:** Schema (column names + types), row count, sample rows (3), null counts per column. No histograms or distributions — those belong in EDA (Phase 5).

**Query pattern:** One DuckDB connection per file. Use `DESCRIBE SELECT * FROM ...` for schema, `SELECT COUNT(*) FROM ...` for row count, `SELECT * FROM ... LIMIT 3` for sample rows, and a dynamic null count query across all columns.

**Output format:** Displayed via `IPython.display.display()` with pandas DataFrames for schema and sample rows. Null counts shown as a summary table.
