# Phase 15 — UI Design Contract

> Visual and interaction contract for the Jinja2 prediction form template served by FastAPI.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope note:** This is a functional utility tool rendered as pure HTML with inline CSS and a
> `<style>` block — no component library, no npm build step, no Tailwind. The template is a single
> Jinja2 file written to `src/<modelname>/vN/templates/index.html` and served by FastAPI's
> `TemplateResponse`. All styling is self-contained in that one file.

## Input Field Presentation

Source: CONTEXT.md D-01, REQUIREMENTS.md WEB-06, CONTEXT.md `<specifics>`.

### Numeric feature inputs

```html
<div class="field">
  <label for="{name}">{name}</label>
  <input type="number" id="{name}" name="{name}" step="any" placeholder="e.g. {example}" required>
</div>
```

- Label text: the raw feature name from `feature_schema` (no transformation, no title-casing)
- `step="any"` on all numeric inputs (supports float64 and int64 equally)
- `placeholder`: `e.g. {example}` where `{example}` is the `example` value from `deployment_metadata.json`
- `required` attribute on all inputs

### Categorical feature inputs

```html
<div class="field">
  <label for="{name}">{name}</label>
  <select id="{name}" name="{name}" required>
    <option value="" disabled selected>Select {name}</option>
    {% for cat in feature.categories %}
    <option value="{{ cat }}">{{ cat }}</option>
    {% endfor %}
  </select>
</div>
```

- Categories populated from `feature_schema[i].categories` (list of strings written by Phase 13 category scan)
- First option is a disabled placeholder: `Select {name}`
- `required` attribute on all selects

### Field layout

Fields are stacked vertically with 16px gap between each field group (`<div class="field">`). Label sits directly above its input with 8px gap.

## Result Card

Source: CONTEXT.md D-04 — "formatted card with probability bars".

The `<div id="result">` is empty on initial page load. After `fetch()` resolves, inject HTML via `innerHTML`.

### Wrapper structure

```html
<div style="margin-top:24px; padding:16px; border:1px solid #e5e7eb; border-radius:6px; background:#ffffff;">
  ...
</div>
```

### All problem types — prediction value

```html
<p style="font-size:20px; font-weight:600; color:#111827; margin:0 0 8px 0;">
  Prediction: <span style="color:#4f46e5;">{{ value }}</span>
</p>
```

- `{{ value }}` is the raw value from the `/predict` JSON response field `prediction`
- No units, no rounding applied by the template — display as returned by the API

### Classification only — probability table

Rendered below the prediction line when the API response includes a `probabilities` key (object mapping class label → probability float 0–1).

```html
<table style="width:100%; border-collapse:collapse; margin-top:8px;">
  <thead>
    <tr>
      <th style="text-align:left; font-size:12px; font-weight:600; color:#6b7280; padding-bottom:4px;">Class</th>
      <th style="text-align:left; font-size:12px; font-weight:600; color:#6b7280; padding-bottom:4px;">Probability</th>
      <th></th>
    </tr>
  </thead>
  <tbody>
    <!-- one row per class, generated via JS after fetch() -->
  </tbody>
</table>
```

Each probability row (injected via JavaScript string template):

```
<tr>
  <td style="font-size:14px; padding:4px 8px 4px 0; color:#111827; white-space:nowrap;">{label}</td>
  <td style="font-size:12px; padding:4px 8px 4px 0; color:#6b7280; white-space:nowrap;">{pct}%</td>
  <td style="width:100%; padding:4px 0;">
    <div style="width:{pct}%; background:#4f46e5; height:12px; border-radius:2px;"></div>
  </td>
</tr>
```

- `{pct}` is `Math.round(probability * 100)` — integer percentage, no decimal
- Bar width is capped at 100% (no overflow)
- Rows are ordered by descending probability

### Regression / clustering — no probability display

Only the prediction value line is rendered. No table, no bars.

## Error State

Source: CONTEXT.md — "Error handling wording for `/predict` validation errors (Pydantic will auto-generate 422 responses; no custom error UX required)" — but the form must handle non-200 responses gracefully.

When `fetch()` returns a non-200 status, inject into `<div id="result">`:

```html
<div style="padding:8px 16px; border:1px solid #dc2626; border-radius:4px; background:#fff5f5;">
  <p style="font-size:14px; color:#dc2626; margin:0; font-weight:600;">Prediction failed</p>
  <p style="font-size:12px; color:#6b7280; margin:4px 0 0 0;">
    The server returned an error. Check your input values and try again.
    (Status: {status})
  </p>
</div>
```

- `{status}` is the HTTP status code from the response (e.g. 422, 500)
- No raw Pydantic error detail is displayed to the user — the generic message is sufficient for this utility tool
- Error replaces any prior result content in `<div id="result">`

## JavaScript Contract

The form uses a single inline `<script>` block at the bottom of `<body>`. No external JS files.

```javascript
document.getElementById('predict-form').addEventListener('submit', async function(e) {
  e.preventDefault();
  const btn = document.getElementById('submit-btn');
  btn.disabled = true;
  const resultDiv = document.getElementById('result');
  resultDiv.innerHTML = '';

  const formData = new FormData(this);
  const body = {};
  formData.forEach((v, k) => { body[k] = isNaN(v) || v === '' ? v : Number(v); });

  try {
    const res = await fetch('/predict', {
      method: 'POST',
      headers: {'Content-Type': 'application/json'},
      body: JSON.stringify(body)
    });
    const data = await res.json();
    if (!res.ok) {
      // render error card
    } else {
      // render result card (prediction + optional probability table)
    }
  } catch (err) {
    // render network error card
  } finally {
    btn.disabled = false;
  }
});
```

Key contracts:
- Numeric field values are coerced from string to `Number` before JSON serialisation (FastAPI/Pydantic expects numeric types)
- `isNaN(v) || v === ''` guard: if the value is not numeric (categorical string) or empty, send as string
- `fetch()` target is always `/predict` (relative URL — works regardless of host/port)
- `Content-Type: application/json` header is required (FastAPI rejects form-encoded bodies on JSON endpoints)

Network error (fetch throws): inject same error card structure with message "Network error — is the service running?"

## Jinja2 Template Structure

Source: CONTEXT.md — Claude's Discretion on whether template is a separate file or Python string.

Decision: separate file at `src/<modelname>/vN/templates/index.html`. FastAPI's `Jinja2Templates` mounts the `templates/` directory. `app.py` renders via:

```python
templates = Jinja2Templates(directory=Path(__file__).parent / "templates")

@app.get("/", response_class=HTMLResponse)
async def form(request: Request):
    return templates.TemplateResponse("index.html", {
        "request": request,
        "model_name": metadata["model_name"],
        "features": metadata["feature_schema"],
        "problem_type": metadata["problem_type"],
    })
```

Template variables available in `index.html`:

| Variable | Type | Description |
|----------|------|-------------|
| `model_name` | string | Display name for `<h1>` and `<title>` |
| `features` | list of `{name, type, categories, example}` | Drives form field generation |
| `problem_type` | string | `regression`, `classification`, or `clustering` |

The `problem_type` variable is used in the JavaScript result renderer to conditionally show the probability table (`if (data.probabilities && "{{ problem_type }}" === "classification")`).
